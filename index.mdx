Mundial de Clicks is an open-source real-time web app where users click to vote for their national team. Every click is a vote, the global ranking updates live via Server-Sent Events, and the infrastructure is built to survive internet-scale traffic spikes — atomic DragonFly counters, a single Lua script per vote batch, and one global polling loop that scales to thousands of simultaneous viewers.

## How it works

Mundial de Clicks is engineered to absorb massive click bursts without falling over. The core trick is that **no row is written per click**:

<Steps>
  <Step title="User clicks a flag">
    The browser accumulates clicks for 400 ms and sends them as a single batched request — up to 10 votes across 3 countries at once. The UI updates instantly (optimistic state).
  </Step>
  <Step title="Atomic Lua script runs in DragonFly">
    A single Lua script validates the captcha session, applies rate limiting, writes `ZINCRBY` on the ranking sorted set, and increments the per-second CPM bucket — all in one round-trip, zero race conditions.
  </Step>
  <Step title="One poller serializes the world">
    A single `setInterval` loop reads DragonFly once per tick, detects overtakes and milestones, serializes the snapshot to bytes **once**, and fans it out to every connected SSE subscriber. 1 viewer or 50,000 — same cost.
  </Step>
  <Step title="SSE delivers real-time updates">
    Every browser subscribed to `GET /api/stream` receives the updated `WorldSnapshot` as a Server-Sent Event and re-renders the ranking without polling.
  </Step>
</Steps>

## Key features

<CardGroup cols={2}>
  <Card title="Atomic vote processing" icon="bolt">
    One Lua script handles rate limiting, session validation, and vote writing in a single DragonFly round-trip.
  </Card>
  <Card title="Global SSE broadcast" icon="satellite-dish">
    One serialization per tick, shared across all viewers. Viewer count doesn't affect DragonFly load.
  </Card>
  <Card title="Cap PoW captcha" icon="shield-halved">
    Proof-of-work anti-bot protection with progressive difficulty that scales with daily vote count per IP.
  </Card>
  <Card title="In-memory fallback" icon="database">
    Works without DragonFly configured — votes run in-process memory so you can develop locally instantly.
  </Card>
  <Card title="Daily vote limits" icon="calendar">
    Per-IP and per-signed-cookie daily caps enforced atomically inside the same Lua script.
  </Card>
  <Card title="Coolify deployment" icon="server">
    Three services (app + DragonFly + Umami) wired together with environment variables — no code changes needed.
  </Card>
</CardGroup>
